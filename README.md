# Архив №404

Реализованный статический русскоязычный сайт-портфолио на Astro. Интерфейс оформлен как ретрофутуристический технический архив; данные сайта и проекты проходят типизированную проверку во время сборки.

## Требования

- Node.js для запуска `npx`.
- Bun 1.3.14 загружается через `npx`; глобальная установка Bun не требуется.

## Команды

```powershell
# Установить зависимости из bun.lock
npx --yes bun@1.3.14 install

# Запустить режим разработки
npx --yes bun@1.3.14 run dev

# Запустить тесты
npx --yes bun@1.3.14 test

# Проверить Astro и TypeScript
npx --yes bun@1.3.14 run check

# Собрать статический сайт в dist
npx --yes bun@1.3.14 run build

# Просмотреть собранный сайт локально
npx --yes bun@1.3.14 run preview
```

## Контент

- Основные данные, SEO-тексты, услуги, технологии и контакты находятся в `src/data/site.json`.
- Реальные проекты хранятся отдельными Markdown-файлами с типизированным frontmatter в `src/content/projects`. Сейчас опубликована одна правдивая запись «Архив №404» со статусом «В разработке» — без вымышленных результатов или изображений.
- Для изображения кейса укажите локальный путь в поле `image` и при желании точный текст в `imageAlt`. Astro соберёт ленивый адаптивный `<picture>` с AVIF/WebP; без `imageAlt` используется безопасная подпись с названием проекта.
- Поля `email` и `telegram` могут оставаться `null` или пустыми. В этом случае сайт показывает состояние закрытого канала и не создаёт фиктивные ссылки.

После изменения контента запустите тесты и `check`, затем соберите сайт командами из раздела выше.

## SITE_URL и BASE_PATH

Без переменных окружения сборка использует локальные значения `SITE_URL=http://localhost:4321` и `BASE_PATH=/`.

Для корневого сайта `example.github.io`:

```powershell
$env:SITE_URL="https://example.github.io"
$env:BASE_PATH="/"
npx --yes bun@1.3.14 run build
```

Для сайта проекта `example.github.io/sample-repo/`:

```powershell
$env:SITE_URL="https://example.github.io"
$env:BASE_PATH="/sample-repo"
npx --yes bun@1.3.14 run build
```

`SITE_URL` должен быть абсолютным HTTP(S)-адресом без пути; регистр схемы и домена нормализуется. `BASE_PATH` должен быть равен `/` или быть путём с начальным слешем без окружающих пробелов. Если переменная задана неверно, сборка завершается с описанием ошибки вместо скрытой подстановки значения по умолчанию.

## GitHub Pages

Workflow `.github/workflows/deploy.yml` запускается при push в `main` или вручную через `workflow_dispatch`. Он сам вычисляет `SITE_URL` и `BASE_PATH` по владельцу и имени репозитория, затем устанавливает зависимости, запускает тесты, Astro check, сборку и официальный деплой Pages.

Все GitHub Actions закреплены на полных commit SHA с комментариями версий, чтобы обновления внешних тегов не меняли pipeline незаметно.

Чтобы включить публикацию:

1. Откройте в репозитории **Settings → Pages**.
2. В поле **Source** выберите **GitHub Actions**.
3. Отправьте изменения в ветку `main` или запустите workflow вручную на вкладке **Actions**.

> **Ограничение project Pages:** для сайта по адресу `/repository/` поисковые роботы запрашивают только origin-root `/robots.txt`, которым репозиторий проекта управлять не может. Добавьте сгенерированный `/repository/sitemap-index.xml` вручную в Google Search Console, Яндекс Вебмастер или аналогичный инструмент. Для управления корневым `robots.txt` используйте корневой пользовательский сайт `<owner>.github.io` или собственный домен.

Служебная страница обслуживания доступна по маршруту `/maintenance/`. Страница 404 генерируется как `dist/404.html`; обе служебные страницы закрыты от индексации и исключены из sitemap.
